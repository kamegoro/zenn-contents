## はじめに

個人開発で AWS のインフラを Terraform で管理していると、こんな悩みが出てきます。

> 「`terraform apply` のたびに実 AWS を使うのはコストも怖いし、壊してしまうのも嫌だ。ローカルで検証したい」

そこで従来は **LocalStack** を使うのが定番でした。ところが 2026 年 3 月、LocalStack は Community Edition を廃止し、最新版の利用には**アカウント登録と認証トークンが必須**になりました。無料プランは「非商用に限定」という制約もつきます。

代替を探していたところ、**MiniStack** というOSSに出会いました。

試してみると、想像以上に本格的でした。**ECS Fargate・RDS・CloudFront・Secrets Manager** といった本番相当の構成がローカルで動き、しかも LocalStack を使うときに必要だった **`tflocal` コマンドが不要**でした。

この記事では、個人開発アプリ（Next.js フロントエンド + Go バックエンド）の AWS インフラを MiniStack + Terraform でローカル再現した手順と、実際にハマったポイントをまとめます。

## TL;DR

- MiniStack は**無料・MIT ライセンス・サインアップ不要**の LocalStack 互換 AWS エミュレータ
- `providers.tf` にエンドポイントを直書きするだけで**通常の `terraform` コマンドがそのまま動く**（`tflocal` 不要）
- VPC / S3 / CloudFront / ECS Fargate / ALB / RDS / Secrets Manager / ECR の **6 モジュール全部動いた**
- `terraform destroy` で S3 Public Access Block がタイムアウトする既知バグがあるが**回避策あり**

## MiniStack とは

MiniStack は LocalStack の代替として登場した、MIT ライセンスのオープンソース AWS エミュレータです。

https://github.com/ministackorg/ministack

LocalStack との主な違いを整理します。

| | LocalStack（現在） | MiniStack |
| --- | --- | --- |
| 料金 | 有料（Hobby は非商用のみ） | 無料・永続 |
| サインアップ | 必須 | 不要 |
| 認証トークン | 必須 | 不要 |
| ライセンス | 独自 | MIT |

MiniStack のもう 1 つの特徴は、**RDS や ECS を「本物のコンテナ」で動かす**点です。RDS を `terraform apply` すると、ホストの Docker socket を通じて本物の PostgreSQL コンテナが起動します。モックではないので、SQL の動作確認まで含めたローカル検証が可能です。

## 今回作ったアーキテクチャ

個人開発アプリ（[task-canvas](https://github.com/kamegoro/task-canvas)）の AWS インフラとして、以下の構成を Terraform で管理しています。

```
Internet
  ├── CloudFront → S3（フロントエンド資産）
  └── ALB → ECS Fargate（Go バックエンド）
                └── RDS PostgreSQL
                    Secrets Manager（JWT / DB パスワード）
```

Terraform のモジュール構成はこうなっています。

| モジュール | 役割 |
| --- | --- |
| `network` | VPC・サブネット・セキュリティグループ |
| `static-site` | S3 + CloudFront によるフロントエンド配信 |
| `fargate-service` | ECS Fargate + ALB による API 配信 |
| `database` | RDS PostgreSQL |
| `secrets` | Secrets Manager（JWT シークレット・DB パスワード）|
| `ecr` | バックエンドイメージ用 ECR リポジトリ |

これらを `terraform/envs/local` でまとめて MiniStack 向けにワイヤリングし、`terraform/envs/dev` / `stg` / `prod` は実 AWS 向けの骨格として用意しています。

## 環境構築

### 前提ツール

- Docker / Docker Compose（MiniStack の起動）
- AWS CLI（apply 後のリソース確認）
- Terraform（[mise](https://mise.jdx.dev/) でバージョン管理）
- jq

### MiniStack を起動する

`docker-compose.yml` を以下のように書きます。

```yaml
services:
  ministack:
    image: ministackorg/ministack:1.3.63
    container_name: aws-infra-ministack
    ports:
      - "4566:4566"
    environment:
      - PERSIST_STATE=1
      - DOCKER_NETWORK=aws-infrastructure_default
    volumes:
      - "./.localstack/volume/state:/tmp/ministack-state"
      - "./.localstack/volume/s3:/tmp/ministack-data/s3"
      - "/var/run/docker.sock:/var/run/docker.sock"
    healthcheck:
      test: ["CMD", "python3", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:4566/_ministack/health')"]
      interval: 5s
      timeout: 5s
      retries: 20
```

`/var/run/docker.sock` のマウントがポイントです。MiniStack が RDS や ECS のコンテナを**ホスト Docker に直接起動**するため、Docker socket へのアクセスが必要になります。

起動はこのコマンドだけです。ヘルスチェックが通るまで待機してから次のステップに進みます。

```bash
docker compose up -d
until [ "$(docker inspect -f '{{.State.Health.Status}}' aws-infra-ministack 2>/dev/null)" = "healthy" ]; do
  sleep 2
done
echo "MiniStack is ready."
```

## ポイント：tflocal が不要な理由

LocalStack を Terraform と組み合わせるときは、エンドポイントを自動でオーバーライドしてくれる `tflocal` コマンドを使うのが一般的です。

```bash
# LocalStack を使う場合の一般的な書き方
tflocal init
tflocal apply
```

しかし MiniStack は LocalStack と**同じエンドポイント形式**をエミュレートするため、`providers.tf` に直接エンドポイントを書けば通常の `terraform` コマンドがそのまま動きます。

```hcl
# terraform/envs/local/providers.tf

provider "aws" {
  region = "us-east-1"

  # MiniStack へのダミー認証情報（実 AWS には接続しない）
  access_key = "test"
  secret_key = "test"

  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    cloudfront             = "http://localhost:4566"
    cloudwatchlogs         = "http://localhost:4566"
    ec2                    = "http://localhost:4566"
    ecr                    = "http://localhost:4566"
    ecs                    = "http://localhost:4566"
    elasticloadbalancing   = "http://localhost:4566"
    elasticloadbalancingv2 = "http://localhost:4566"
    iam                    = "http://localhost:4566"
    rds                    = "http://localhost:4566"
    s3                     = "http://s3.localhost.localstack.cloud:4566"
    secretsmanager         = "http://localhost:4566"
  }
}
```

`s3` だけ専用のエンドポイント（`s3.localhost.localstack.cloud:4566`）を指定している点に注意してください。`s3.localhost.localstack.cloud` は公開登録済みのドメインで、DNS 解決すると `127.0.0.1` に返ります（`/etc/hosts` の編集は不要です）。MiniStack は S3 リクエストをこのドメイン経由で受け付けており、他のサービスと同じ `localhost:4566` では動作しません。

`s3_use_path_style = true` は、バケット名を URL のパス部分（`http://s3.localhost.localstack.cloud:4566/<bucket-name>/<key>`）に含める path-style アドレッシングを強制する設定です。

この `providers.tf` を `terraform/envs/local/` に置き、`envs/dev` 以降は通常の `provider "aws" {}` を書けば、**同じモジュールを実 AWS とローカルの両方で使い回せます**。

## terraform apply してみる

ディレクトリ構成はこうなっています。

```
terraform/
  modules/
    network/          # VPC・サブネット・セキュリティグループ
    static-site/      # S3 + CloudFront
    fargate-service/  # ECS Fargate + ALB
    database/         # RDS PostgreSQL
    secrets/          # Secrets Manager
    ecr/              # ECR リポジトリ
  envs/
    local/            # MiniStack 向け（この記事のメイン）
    dev/              # 実 AWS 向け（開発環境）
    stg/              # 実 AWS 向け（ステージング）
    prod/             # 実 AWS 向け（本番）
```

apply の手順はシンプルです。

```bash
cd terraform/envs/local

terraform init
terraform plan
terraform apply
```

apply が完了したら、AWS CLI でリソースを確認できます（`--endpoint-url` で MiniStack を向きます）。

```bash
# VPC 確認
aws --endpoint-url=http://localhost:4566 ec2 describe-vpcs \
  --query 'Vpcs[].VpcId' --output text

# RDS インスタンス確認
aws --endpoint-url=http://localhost:4566 rds describe-db-instances \
  --query 'DBInstances[].DBInstanceIdentifier' --output text

# ECS クラスター確認
aws --endpoint-url=http://localhost:4566 ecs list-clusters \
  --query 'clusterArns' --output text

# S3 バケット確認
aws --endpoint-url=http://s3.localhost.localstack.cloud:4566 s3 ls
```

`terraform output` でエンドポイント一覧もまとめて確認できます。

```bash
terraform output
# alb_dns_name             = "..."
# db_endpoint              = "..."
# db_name                  = "..."
# ecr_repository_url       = "..."
# frontend_bucket_name     = "..."
# ecs_cluster_name         = "..."
```

## ハマったところ

### S3 Public Access Block の destroy がタイムアウトする

`terraform destroy` を実行すると、`aws_s3_bucket_public_access_block` の削除でタイムアウトします。

```
Error: waiting for S3 Bucket Public Access Block deletion: timeout while waiting for state to become 'destroyed'
```

これは MiniStack v1.3.63 の既知バグです。`DeletePublicAccessBlock` は成功を返すものの、`GetPublicAccessBlock` が古い設定を返し続けるため、Terraform の待機ループがタイムアウトします。なお、**v1.3.64（2026年6月15日リリース）で修正済み**のため、最新版を使う場合はこの回避策が不要です。

https://github.com/ministackorg/ministack/issues/915

**回避策**（v1.3.63 を使う場合）：destroy 前にこのリソースだけ state から外します。

```bash
cd terraform/envs/local

terraform state rm module.static_site.aws_s3_bucket_public_access_block.frontend
terraform destroy
```

Makefile に組み込んでおくと、毎回意識しなくて済みます。

```makefile
tf-destroy:
	cd terraform/envs/local && \
	terraform state rm module.static_site.aws_s3_bucket_public_access_block.frontend 2>/dev/null; \
	terraform destroy
```

### Secrets Manager の apply が 2 回目以降に失敗する

`apply` → `destroy` を繰り返すと、2 回目の apply でこのエラーが出ます。

```
ResourceExistsException: A secret with this name already exists.
```

Secrets Manager はデフォルトでシークレット削除後 **30 日間のリカバリー期間**があり、この期間中は同名シークレットを再作成できません。

**回避策**：`recovery_window_in_days = 0` を設定して即時削除にします。

```hcl
resource "aws_secretsmanager_secret" "this" {
  name                    = var.name
  recovery_window_in_days = 0  # apply/destroy を繰り返すためのローカル環境向け設定
}
```

本番環境では削除後の誤復元を防ぐためリカバリー期間は残しておき、`envs/local` だけ 0 にするのがよいです。

## MiniStack の制約まとめ

今回確認した制約も正直に書いておきます。

| 制約 | 内容 |
| --- | --- |
| CloudFront のコンテンツ配信 | コントロールプレーンのみ。`*.cloudfront.net` 経由の配信は動かない |
| S3 Public Access Block destroy | v1.3.63 の既知バグ（v1.3.64 で修正済み・回避策あり）|

CloudFront 経由のコンテンツ確認は、S3 バケットへ直接アクセスする形で代替できます。

```bash
aws --endpoint-url=http://s3.localhost.localstack.cloud:4566 \
  s3 cp s3://$(terraform -chdir=terraform/envs/local output -raw frontend_bucket_name)/index.html -
```

## おわりに

MiniStack + Terraform を使い、個人開発の AWS インフラ（VPC / ECS Fargate / RDS / CloudFront / Secrets Manager / ECR）をローカルで再現できました。

特に印象的だったのは **`tflocal` が不要**な点です。`providers.tf` にエンドポイントを直書きするだけで動いたのは、LocalStack ユーザーにとって地味に嬉しいポイントだと思います。

制約はいくつかありますが、個人開発でのローカル検証目的であれば十分実用的です。「`terraform apply` のたびに実 AWS を触るのが怖い」という状況が、これで大きく改善されました。
