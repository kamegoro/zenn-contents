## はじめに

個人開発で AWS のインフラを Terraform で管理していると、こういう悩みが出てきます。

> 「`terraform apply` のたびに実 AWS を使うのはコストが怖いし、壊すのも嫌だ。ローカルで試したい」

そこで従来は **LocalStack** を使うのが定番でした。ところが 2026 年 3 月、LocalStack は Community Edition を廃止し、最新版の利用には**アカウント登録と認証トークンが必須**になりました。無料の Hobby プランも「非商用に限定」という制約つきです。

代わりになるものを探していたところ、**MiniStack** という OSS に出会いました。試してみたら、**ECS Fargate・RDS・CloudFront・Secrets Manager** といった構成がローカルで動きました。

この記事では、個人開発アプリの AWS インフラを MiniStack + Terraform でローカル再現した手順をまとめます。

## TL;DR

- MiniStack は**無料・MIT ライセンス・サインアップ不要**の LocalStack 互換 AWS エミュレータ
- `providers.tf` にエンドポイントを書くだけで**通常の `terraform` コマンドがそのまま使える**
- VPC / S3 / CloudFront / ECS Fargate / ALB / RDS / Secrets Manager / ECR が**ローカルで動いた**
- Makefile に操作をまとめておくと **`make e2e` で起動から E2E テストまで一発で回せる**

## MiniStack とは

MiniStack は LocalStack の代替として登場した、MIT ライセンスのオープンソース AWS エミュレータです。

https://github.com/ministackorg/ministack

LocalStack との違いを表にまとめます。

| | LocalStack（現在） | MiniStack |
| --- | --- | --- |
| 料金 | 有料（Hobby は非商用のみ） | 無料 |
| サインアップ | 必須 | 不要 |
| 認証トークン | 必須 | 不要 |
| ライセンス | 独自 | MIT |

個人的に面白いと思ったのは、**RDS や ECS を「本物のコンテナ」で動かす**点です。`terraform apply` で RDS を作ると、ホストの Docker socket を通じて本物の PostgreSQL コンテナが起動します。モックではないので、SQL の動作確認までローカルでできるのは嬉しいところです。

## 今回作ったアーキテクチャ

個人開発アプリ（[task-canvas](https://github.com/kamegoro/task-canvas)）の AWS インフラとして、以下の構成を Terraform で管理しています。

```
Internet
  ├── CloudFront → S3（フロントエンド資産）
  └── ALB → ECS Fargate（Go バックエンド）
                └── RDS PostgreSQL
                    Secrets Manager（JWT / DB パスワード）
```

Terraform のモジュール構成はこうなっています。

| モジュール | 役割 |
| --- | --- |
| `network` | VPC・サブネット・セキュリティグループ |
| `static-site` | S3 + CloudFront によるフロントエンド配信 |
| `fargate-service` | ECS Fargate + ALB による API 配信 |
| `database` | RDS PostgreSQL |
| `secrets` | Secrets Manager（JWT シークレット・DB パスワード）|
| `ecr` | バックエンドイメージ用 ECR リポジトリ |

これらを `terraform/envs/local` でまとめて MiniStack 向けにワイヤリングし、`terraform/envs/dev` / `stg` / `prod` は実 AWS 向けとして分けています。**同じモジュールを実 AWS とローカルの両方で使い回す**構成です。

## 環境構築

### 前提ツール

- Docker / Docker Compose
- AWS CLI
- Terraform（[mise](https://mise.jdx.dev/) でバージョン管理）
- jq

### MiniStack を起動する

`docker-compose.yml` はこんな感じです。

```yaml
services:
  ministack:
    image: ministackorg/ministack:1.3.64
    container_name: aws-infra-ministack
    ports:
      - "4566:4566"
    environment:
      - PERSIST_STATE=1
      - DOCKER_NETWORK=aws-infrastructure_default
    volumes:
      - "./.localstack/volume/state:/tmp/ministack-state"
      - "./.localstack/volume/s3:/tmp/ministack-data/s3"
      - "/var/run/docker.sock:/var/run/docker.sock"
    healthcheck:
      test: ["CMD", "python3", "-c", "import urllib.request; urllib.request.urlopen('http://localhost:4566/_ministack/health')"]
      interval: 5s
      timeout: 5s
      retries: 20
```

`/var/run/docker.sock` のマウントがポイントです。MiniStack が RDS や ECS のコンテナを**ホスト側の Docker に直接起動する**ため、Docker socket へのアクセスが必要になります。

### providers.tf で MiniStack に接続する

`terraform/envs/local/providers.tf` にエンドポイントを書くだけで、通常の `terraform` コマンドがそのまま使えます。

```hcl
# terraform/envs/local/providers.tf

provider "aws" {
  region = "us-east-1"

  access_key = "test"
  secret_key = "test"

  s3_use_path_style           = true
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    cloudfront             = "http://localhost:4566"
    cloudwatchlogs         = "http://localhost:4566"
    ec2                    = "http://localhost:4566"
    ecr                    = "http://localhost:4566"
    ecs                    = "http://localhost:4566"
    elasticloadbalancing   = "http://localhost:4566"
    elasticloadbalancingv2 = "http://localhost:4566"
    iam                    = "http://localhost:4566"
    rds                    = "http://localhost:4566"
    s3                     = "http://s3.localhost.localstack.cloud:4566"
    secretsmanager         = "http://localhost:4566"
  }
}
```

`access_key` / `secret_key` は `"test"` で大丈夫です。MiniStack はダミーの認証情報を受け付けるので、実 AWS のクレデンシャルは不要です。

`s3` だけエンドポイントが `s3.localhost.localstack.cloud:4566` になっています。`s3.localhost.localstack.cloud` は公開登録されたドメインで、DNS が `127.0.0.1` を返します（`/etc/hosts` の編集は不要です）。MiniStack は S3 のリクエストをこのドメイン経由で受け付ける仕組みになっていて、他のサービスと同じ `localhost:4566` では動きません。

`s3_use_path_style = true` は、バケット名を URL のパス部分に含める path-style アドレッシングを強制する設定です。

`envs/dev` 以降は通常の `provider "aws" {}` を書くだけなので、モジュール側は環境を意識しない作りになります。

## ローカル環境向けの Terraform 設定

`envs/local/main.tf` には、ローカルで apply/destroy を繰り返すための設定がいくつか必要でした。

```hcl
# terraform/envs/local/main.tf

module "static_site" {
  source = "../../modules/static-site"

  name        = var.name
  bucket_name = var.frontend_bucket_name

  force_destroy = true
}

module "secrets" {
  source = "../../modules/secrets"

  name        = var.name
  db_password = module.database.password

  recovery_window_in_days = 0
}

module "fargate_service" {
  source = "../../modules/fargate-service"

  # ... 省略 ...

  secrets_as_environment = true
}
```

それぞれ補足します。

**`force_destroy = true`（S3）**

S3 バケットにオブジェクトが残っている状態で `terraform destroy` すると、デフォルトでは削除に失敗します。ローカル環境ではフロントエンドの資産をアップロードしてからテストする流れになるため、`force_destroy = true` でバケットごと強制削除できるようにしています。

**`recovery_window_in_days = 0`（Secrets Manager）**

Secrets Manager はデフォルトで削除後 **30 日間のリカバリー期間**があり、この期間中は同じ名前のシークレットを再作成できません。`0` にすると即時削除になるので、apply/destroy を繰り返してもエラーになりません。本番環境ではリカバリー期間を残しておくべきです。

**`secrets_as_environment = true`（ECS Fargate）**

ECS タスク定義の `secrets`（Secrets Manager の ARN を指定して値を注入する機能）は、MiniStack では動かないケースがありました。シークレットの値を通常の環境変数として渡す回避策を用意しています。

## terraform apply してみる

```bash
cd terraform/envs/local

terraform init
terraform plan
terraform apply
```

apply が完了したら、AWS CLI でリソースが作られているか確認できます。`--endpoint-url` で MiniStack を向けるだけです。

```bash
# VPC
aws --endpoint-url=http://localhost:4566 ec2 describe-vpcs \
  --query 'Vpcs[].VpcId' --output text

# サブネット
aws --endpoint-url=http://localhost:4566 ec2 describe-subnets \
  --query 'Subnets[].[SubnetId, CidrBlock, AvailabilityZone]' --output table

# RDS インスタンス
aws --endpoint-url=http://localhost:4566 rds describe-db-instances \
  --query 'DBInstances[].{ID: DBInstanceIdentifier, Engine: Engine, Status: DBInstanceStatus}' --output table

# ECS クラスター
aws --endpoint-url=http://localhost:4566 ecs list-clusters \
  --query 'clusterArns' --output text

# S3 バケット
aws --endpoint-url=http://s3.localhost.localstack.cloud:4566 s3 ls

# Secrets Manager
aws --endpoint-url=http://localhost:4566 secretsmanager list-secrets \
  --query 'SecretList[].Name' --output text
```

RDS の場合、`docker ps` で PostgreSQL コンテナが実際に起動しているのも確認できます。

```bash
docker ps --filter "ancestor=postgres" --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

本物の PostgreSQL が動いているので、`psql` で直接つないで SQL を流すこともできます。

## Makefile で操作をまとめる

MiniStack の起動・apply・destroy を毎回手打ちするのは面倒なので、Makefile にまとめています。

```makefile
TF_DIR := terraform/envs/local

up:
	docker compose up -d
	@until [ "$$(docker inspect -f '{{.State.Health.Status}}' aws-infra-ministack 2>/dev/null)" = "healthy" ]; do sleep 2; done
	@echo "MiniStack is ready."

down:
	docker compose down

tf-apply:
	cd $(TF_DIR) && terraform apply

tf-destroy:
	cd $(TF_DIR) && terraform destroy

sync-frontend:
	./scripts/sync-frontend.sh $(FRONTEND_DIR)
```

### S3 への静的アセットアップロード

フロントエンドのビルド成果物を MiniStack の S3 にアップロードするスクリプトも用意しています。

```bash
# scripts/sync-frontend.sh（抜粋）

# MiniStackはAWS CLIのデフォルトのチェックサムアルゴリズム(CRC64NVME)を
# サポートしていないため、WHEN_REQUIREDに変更する
export AWS_REQUEST_CHECKSUM_CALCULATION="${AWS_REQUEST_CHECKSUM_CALCULATION:-WHEN_REQUIRED}"

aws --endpoint-url=http://s3.localhost.localstack.cloud:4566 \
  s3 sync "$FRONTEND_DIR" "s3://${BUCKET_NAME}" --delete
```

MiniStack は AWS CLI のデフォルトのチェックサムアルゴリズム（CRC64NVME）をサポートしていないため、`AWS_REQUEST_CHECKSUM_CALCULATION=WHEN_REQUIRED` を設定する必要があります。これを知らないと `s3 sync` でエラーになります。

### make e2e で一括実行

最終的に、MiniStack の起動から E2E テストまでを一発で回せる `e2e` ターゲットも作りました。

```makefile
e2e: up
	cd $(TF_DIR) && terraform apply -auto-approve
	@if [ -n "$(FRONTEND_DIR)" ]; then $(MAKE) sync-frontend FRONTEND_DIR=$(FRONTEND_DIR); fi
	@status=0; \
	if [ -n "$(E2E_CMD)" ]; then \
		( set -a; source <(./scripts/tf-outputs-env.sh); set +a; cd $(E2E_DIR) && $(E2E_CMD) ) || status=$$?; \
	fi; \
	$(MAKE) tf-destroy TF_DESTROY_ARGS=-auto-approve; \
	$(MAKE) down; \
	exit $$status
```

```bash
make e2e FRONTEND_DIR=../task-canvas/frontend/out E2E_DIR=../task-canvas-e2e E2E_CMD="mvn test"
```

`tf-outputs-env.sh` が Terraform の output を `TF_OUT_*` 形式へ変換するため、E2E テスト側は接続先を環境変数から読み取るだけです。テストの結果によらず、最後は destroy と down でクリーンアップされます。

## おわりに

MiniStack + Terraform を使って、個人開発の AWS インフラ（VPC / ECS Fargate / RDS / CloudFront / Secrets Manager / ECR）をローカルで再現してみました。

`terraform apply` するたびに実 AWS を触る怖さがなくなったのと、Makefile にまとめておけば `make e2e` で起動からテストまで一発で回せるのが気に入っています。
